# A2 站页需求实测分析（2026-06-12）

> SEO 调查报告（`../seo-geo-research-2026.md`）行动项 A2 的执行结果。
> 回答两个问题：① 站级 vs 区/市级，流量盘子差多少？② 1831 站值得建页的砍线在哪？

## 方法（全部免费数据源，三角验证）

| 数据源 | 覆盖 | 测什么 |
|---|---|---|
| ja.Wikipedia 条目浏览量（12 个月：2025-06〜2026-05） | **全量 1831 站**（1819 有条目）+ 23 区 + 16 市 | 关注度总排名 / 分布形态 |
| aramakijake.jp 月间搜索量估计 | 26 个校准关键词 | 浏览量 → 「住みやすさ」绝对搜索量的换算锚点 |
| Google 联想词（「{名} 住」） | 分层抽样 75 站 + 23 区 | 居住意图查询的真实存在性 |

原始数据同目录 4 个 json。脚本一次性（`C:/tmp/kayoha-demand/`，未入库）。

## 发现 1：区级查询的心智是「行政手续」，不是居住决策

- 23 区的「{区名} 住」联想词**全部被「住民票/住民税」霸占**，「住みやすさ」仅 14/23 个区进得了联想列表且排位靠后
- 校准数据：**区级「住みやすさ」月搜索量（168–384）≈ 单个头部站（312–472）**。世田谷区 168/月 < 中野站 472/月
- 而每个区含 10–40 个站，每站有自己的查询族 → **站级总盘子比区级大一个数量级**

→ **用户「换方向做区/市报告」的方案不能替代站页**：50 个区市页的总查询盘子 ≈ 20 个头部站，撑不起旺季几千的目标。但区市页作为**内链 hub 层**仍然要做（约 39 页，吃区级查询 + 解决站页抓取存留）。

## 发现 2：站级需求衰减比预想平缓，砍线不在 300，在 600–800

12 个月 Wikipedia 浏览量分布（1819 站）：

| 阈值（年浏览） | 站数 |
|---|---|
| ≥10 万 | 8 |
| ≥5 万 | 33 |
| ≥3 万 | 160 |
| ≥2 万 | 357 |
| ≥1 万 | 957（中位数 = 1.05 万） |
| ≥5 千 | 1521 |

校准锚点（年浏览 → 「住みやすさ」月搜，Google）：吉祥寺 41k→384、高円寺 21k→312、船堀 13k→208、雑色 14k→72、西大島 6k→112、東長崎 13k→0。**中尾部站仍有真实搜索量**（联想词命中率：头部 92%、中部 72%、尾部 36%）。

## 发现 3：砍线模拟（量级参考，不承重）

假设：查询族 = 住みやすさ×3（家賃相場实测是其 1.5–2 倍 + 一人暮らし等）；排名成功后综合 CTR 10%（已含 AIO 截流）。

| 砍线 | 成熟期日均点击（模型值） |
|---|---|
| top 100 | ~175 |
| top 300 | ~440 |
| **top 600** | **~740** |
| top 800 | ~930 |
| 全量 1831 | ~1400 |

旺季（1–3 月）搜索量约 ×1.5–2 → **top 600 在旺季对应 1100–1500/天，top 800–1200 才稳进「几千」区间**。
⚠️ 模型最大不确定性是「排名成功率」（新域名 vs スマイティ等），所以必须分批 + 设观测门槛，不能一次铺满。

## 建议方案（金字塔 + 数据门控分批）

```
第 0 层  区市 hub 页 39 页（23 区 + 16 市）       — 内链中枢 + 吃区级查询
第 1 批  top 150 站（浏览量 × 联想词命中 复合排序）— 试点：验证收录率/排名/参与度
第 2 批  → top 600                                — 第 1 批数据达标后扩
第 3 批  → top 1200                               — 第 2 批达标后再扩（边际收益模型值仍正）
不建页   余下 ~600 长尾站                          — 数据留在地图应用内
```

观测门槛（每批 4–6 周后）：收录率 ≥70%、GSC 展示量持续上升、零点击页面占比 <30%。不达标 → 停下修模板，不扩量。

排序注意：浏览量 top 里的東京/新宿/高輪ゲートウェイ等是「交通知名度」不是居住需求（東京駅 住みやすさ 没人搜）——复合排序要用「联想词命中 + 住宅属性」加权，30 个 /to 通勤地与站页的意图区分在模板设计（A1）时处理。

## 砍线结论

**立项规模：39 区市页 + 最终 600–1200 站页（分 3 批、数据门控），首批 150。**
1831 全量不必要（用户直觉正确），区级替代也不成立（盘子差一个数量级）——金字塔两层都要，主力在站层。
