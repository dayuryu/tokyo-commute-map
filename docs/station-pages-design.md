# 站页工程设计书（A1）— 駅別住みやすさページ

> 2026-06-12。依据：`docs/seo-geo-research-2026.md`（GEO 调查）+ `docs/research/station-demand-analysis-2026-06.md`（A2 实测）。
> 目标：旺季（1–3 月）几千 UV/天的主力流量资产。倒排 2027 年 1 月：Q3 首批 150 上线，Q4 扩量。

---

## 1. 信息架构（金字塔三层）

```
/area               区市索引（1 页）        ← HeaderMenu / footer 内链入口
/area/{ward}        区市 hub（39 页）       ← 23 区 + 16 市，吃区级查询 + 内链中枢
/st/{slug}          站页（首批 150 → 1200）  ← 主力，吃「{駅名} 住みやすさ/家賃」查询族
```

- **只做 ja**。目标查询全部是日语；zh/en 的 area_features 翻译虽存在，但多语版会把首批页面数 ×3 而流量贡献近零，留待站页被验证后再议。canonical 自指、不出 hreflang。
- slug：罗马字（`station-slugs.json` 为 SSOT，1831 全量唯一）。两个代码兜底 slug（`asakusa-2100201`→`asakusa`、`oyama-100404`→`oyama`，因同名组里另一站需求极低，让高需求站占干净 slug）上线前手工修正。
- 与 `/to/{slug}`（通勤地视角）意图不同：/to = 「去那里上班，住哪」，/st = 「住这里怎么样」。30 个重叠站（新宿等）两页并存，互链，不互蚕食（查询意图不同）。

## 2. 站页模板（防 scaled content 设计）

每个 section 标注【独有数据 = 不可为 null 字段】，这是对抗「同模板换地名」判定的核心（调查 F3/F4）：

| # | Section | 内容 | 独有性 |
|---|---|---|---|
| 1 | **答案优先开头**（GEO 核心） | 2-3 句直答：「{駅}から新宿までは最短 {n} 分、家賃相場は 1R {x} 万円。{一句站格特征}」 | 数据注入 + 文案变体 |
| 2 | **通勤マトリクス表** | 到 30 通勤地的实算分钟数，升序、≤30 分高亮，表 caption 带数据基准日 | 🏆 全网独有（スマイティ无此数据），每站数值天然不同 |
| 3 | 家賃相場 | 政府统计（95% 站覆盖）+ SUUMO 101 站按间取り，与「30 分圏平均」对比一句 | 数据 |
| 4 | **街の特徴**（编辑性正文） | **150–250 字/站，外部 LLM 批量初稿 + 运营过目**（用 area_features 50-75 字做种子扩写，流水线复用 area_features 的 47 批工作流） | 每站独立文本 = 独特内容比的承重墙 |
| 5 | 隣駅ナビ | 同沿线前后站对比（通勤 ±分、家賃 ±万円）+ 链接（邻接关系从 Dijkstra 图数据派生） | 数据 + 内链网 |
| 6 | 似た駅 | 通勤画像 × 家賃相近的 4-6 站（build 时计算）+ 链接 | 数据 + 内链网 |
| 7 | FAQ（4 问） | 数据生成：「{駅}から{最重要通勤地}まで何分？」「家賃相場は？」「終電は何時？（无数据则不出）」「どんな人に向いている？（街特征派生）」+ FAQPage JSON-LD | 答案全部数据注入 |
| 8 | CTA | 「地図でこの駅を見る」（deep link 选中本站）+ AI 推薦入口 | — |
| 9 | コミュニティ評価 | client island 拉 Supabase（非 SEO 内容，不入 SSG HTML，有则显示） | — |

- **独特内容比自查**：2+3+4+5+6+7 合计中，纯模板文字控制在 ~50-60%，数据值 + 编辑文 + FAQ 答案 ≥40%——高于调查 F4 的 30-40% 经验线。
- JSON-LD：`TrainStation`（geo 坐标 + address）+ `BreadcrumbList`（Top → /area/{ward} → 本页）+ `FAQPage`。
- 通勤矩阵用语义化 `<table>`，数字可被 AI 直接摘录（GEO 战术 B1 一并落地）。

## 3. 区市 hub 模板（39 页）

- 本区市车站一览表（站名 / 最重要通勤地分钟 / 家賃，每行链 /st/）
- 数据排行模块：「本区通勤最快 top5」「家賃最安 top5」（数据生成，每区不同）
- 区级生活总述 200–300 字（39 页量小，外部 LLM 初稿 + 运营过目）
- 注意 A2 发现：区级查询里「住みやすさ」竞争不过「住民票」——标题按「{区名}の住みやすさ・家賃相場【駅別データ】」打数据差异化
- JSON-LD：CollectionPage + ItemList + BreadcrumbList

## 4. 内链网（防 130 天失索引，调查 F5）

```
HeaderMenu/Welcome footer → /area → 39 hub → 全部站页
站页 ↔ 隣駅（沿线网）  站页 ↔ 似た駅（数据网）  站页 ↔ 所属 hub（回链）
/to/{slug} ←→ /st/{slug}（30 个重叠站互链）
sitemap：/area + 39 + 批次内站页（lastmod 用数据基准日，不用 build 时间）
```

更新信号：通勤数据/家賃数据年度更新时全量改 lastmod；评价数为动态内容不影响。

不达标 → 停止扩量，修模板再观测。其余 ~600 长尾站不建页。

## 6. 实装顺序

1. **数据准备**：邻接关系（从图数据派生）、似た駅计算、slug 手工修正 2 件、批次清单定稿
2. **模板实装**：/st/[slug] + /area/[ward] + /area，SSG（generateStaticParams 只生成批次内 slug）
3. **文案生产**：150 站 × 200 字 + 39 区市 × 250 字，外部 LLM 批量（复用 area_features prompts→merge 工作流）→ 运营过目
4. **上线**：sitemap + HeaderMenu/footer 内链 + 首批索引请求（Search Console 配额排程）
5. **观测**：GSC 批次 dashboard + GA4（B2 的 AI referral 分组一并配）

## 7. 已决事项（蒸し返さない）

- 区市报告**不替代**站页（A2 实测：盘子差一个数量级、区级心智是行政手续）
- 1831 全量不做，最终 600–1200，数据门控
- 非住宅 13 站黑名单沿用 AI 推荐的 `NON_RESIDENTIAL_STATION_CODES`
- 多语不做（ja only），X カード不做
