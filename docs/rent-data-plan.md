## 採用方針

### Phase 1（2026-05-13 実装完了、当日 3 回改訂）— 101 駅（通勤先 30 + 住宅地 71）の SUUMO 駅別相場

- **データ源**: SUUMO 駅別賃料相場ページ `https://suumo.jp/chintai/soba/<pref>/ek_<ek_id>/`
- **生データ条件**: 新築マンション、駅徒歩 1〜5 分以内（SUUMO 独自集計、相場の上限寄り）
- **UI 表示条件**: 徒歩 10 分以内目安（生データ × walk_factor=0.9 で推算）
- **取得日**: 2026-05-13
- **データ基準日**: 2026-04-13
- **収録項目**: 1R / 1K / 1DK / 1LDK / 2LDK / 3LDK の平均賃料（万円）
- **収録駅構成**:
  - **通勤先 30 駅**（`destinations.ts` と同期、オフィス・ビジネス街）
  - **住宅地 71 駅**：
    - 東京 23 区西部：阿佐ヶ谷・荻窪・西荻窪・高円寺・東中野・代々木・原宿・恵比寿・広尾・麻布十番・白金高輪・青山一丁目・学芸大学・都立大学・田園調布・駒沢大学・桜新町・用賀・成城学園前・経堂・千歳船橋・明大前・千歳烏山・調布・練馬・石神井公園 + 既存（中野・中目黒・自由が丘・三軒茶屋・二子玉川・下北沢）
    - 東京 23 区東部・北部：月島・勝どき・清澄白河・両国・錦糸町・亀戸・王子・赤羽・巣鴨・駒込 + 既存（北千住）
    - 多摩地区：三鷹・武蔵境・府中・国分寺・町田
    - 神奈川：日吉・元住吉・新丸子・鷺沼・たまプラーザ・青葉台・新横浜・上大岡・関内・鶴見 + 既存（武蔵小杉・横浜・みなとみらい・川崎）
    - 埼玉：川口・蕨・所沢・川越・和光市 + 既存（大宮・浦和）
    - 千葉：船橋・市川・津田沼・本八幡・南行徳 + 既存（千葉）
- **格納先**: `public/data/manual_rent_data.json`
- **アクセス**: `lib/manual-rent.ts` の `loadManualRentData()` / `getSingleRentLabel()` / `getCoupleRentLabel()`（換算は formatter 側で自動適用）
- **UI**: `StationDrawer` の「家賃目安」DetailRow に `11.2 万円〜 · 1LDK 20.4 万円` 形式で表示、hint に「徒歩 10 分以内目安」明記

#### 換算戦略（5 分 → 10 分）

SUUMO は新築・徒歩 5 分以内のデータしか公開していないが、本サイトのユーザーは「実際に住める家」を探しているため、5 分以内の上限寄り相場は過大評価となる。不動産業界共識として徒歩 10 分以内の相場は 5 分以内より約 8-12% 低いため、**walk_factor = 0.9** を採用。

- JSON 内には生データ（5 分）を保持し、`lib/manual-rent.ts` の `applyFactor()` で UI 表示時に乗算
- 将来 Phase 2 で政府データ baseline が入った時に factor を逆算で校正できる疎結合設計
- `manual_rent_data.json` の `_meta.walk_factor` で値を一元管理、UI 表示はその値を反映

#### 法的観点

- **scraping ではなく人間/AI による単点目視**: WebFetch で各駅 1 回ずつ、合計 30 回のアクセス。listing データの大量取得ではなく、SUUMO が同ページに公開している中央値の数字を読み取った
- **アフィリエイト提携 ≠ データ利用権**: A8 / SUUMO 提携は誘導リンクの権利のみ、データの再利用は契約範囲外。ただし「公開ページの中央値を目安として参照」は通常閲覧の範疇
- **景表法対応**: hint に「目安」と「算出条件」を明記、SUUMO 出典は `_meta` に記録（必要に応じて UI にも露出可能）

#### 主な落とし穴

- **東京駅 1K = 23.1 万円の outlier**: SUUMO 集計の母数が少ない物件タイプで起きる偏り。`_meta.note_1k_tokyo_outlier` に記載済み。UI 側で smoothing するか主人判断
- **新築・徒歩 5 分条件のバイアス**: 相場の上限寄りになるため、ユーザーが実際に契約する家賃より高めに見える。hint 表記で開示する以上の対応は次フェーズで検討

#### メンテナンス運用

- SUUMO 相場ページは月次更新される（実データ基準日が月内に進む）
- 推奨頻度: 四半期ごとに `manual_rent_data.json` を再生成
- 自動化案: `scripts/refresh_manual_rent.py`（未実装、将来 GitHub Actions で月次走らせる）

### Phase 2（2026-05-13 実装完了）— 全関東駅 1940/2043 (95%) を政府データでカバー

- **データ源**: 総務省統計局「住宅・土地統計調査」（令和5年 = 2023 年）
- **統計表 ID**: `0004021452`「居住室の畳数(6区分)別住宅の１か月当たり家賃(借家)—全国、都道府県、市区町村」
- **抽出条件**: cat01=0（全畳数の総数）× cat02=2（家賃 0 円除外、社宅・公営除く民間賃貸ベース）
- **粒度**: 市区町村（区・市・町・村、人口 1.5 万人以上）
- **データ取得**: e-Stat API（v3.0、appId 必要、無料）
- **カバレッジ**: 関東 8 都県 2043 駅のうち **1940 駅（95.0%）**
- **未カバー 5%**: 人口 1.5 万人未満の小町村（奥多摩町・真鶴町・越生町・美里町等）で e-Stat が個別データを公開していない

#### 生成スクリプト

- `scripts/build_government_rent_data.py` — e-Stat API → 305 関東市区町村の家賃データ
- `scripts/build_station_government_rent.py` — station.csv の address 解析 → 駅 code への join
- 出力先:
  - `public/data/government_rent_data.json`（305 市区町村、参考用）
  - `public/data/station_government_rent.json`（1940 駅 → 月家賃 円、UI 用）

#### マッピング戦略（address → 市区町村）

1. address から「都道府県」prefix を剥がす
2. 「○○郡」prefix を剥がす（郡レベルは政府データに無し）
3. 政令市判定: 「さいたま市/千葉市/横浜市/川崎市/相模原市」+「△△区」を末端に
4. 一般: lookup-driven match（短い候補から area_lookup でヒット判定、最初の命中を採用）
   - 「東村山市」は「東村」ハズレ後「東村山市」で命中
   - 「市川市」は「市」/「市川」/「市川市」と試行、命中で停止

#### UI 統合（StationDrawer 家賃 DetailRow）

二層 fallback で表示:
- **Layer 1 (SUUMO 101 駅)**: `11.2 万円〜 · 1LDK 22.7 万円` 形式、hint「徒歩 10 分以内目安（SUUMO 由来）」
- **Layer 2 (政府 1940 駅)**: `区平均 8.5 万円` 形式、hint「住宅・土地統計調査（区平均）」
- **Layer 3 (なし)**: `—`、hint「データなし（人口 1.5 万未満）」

#### lib

- `lib/government-rent.ts` — `loadGovernmentRentData()` / `formatGovernmentRent()` / 型定義
- `lib/manual-rent.ts` — Phase 1 SUUMO 用、変更なし

#### 注意事項（SUUMO vs 政府データの baseline 差）

両者は性質が異なる：
- **SUUMO**: 新築・徒歩 5 分以内（× 0.9 で 10 分換算）、相場上限寄り
- **政府**: 全築年数・全立地、市区町村全体の借家中央値、地区全体感

例：渋谷区
- SUUMO（渋谷駅 1R）: 17.9 × 0.9 = **16.1 万円〜**（新築駅近の 1R）
- 政府（渋谷区平均）: **13.1 万円**（全築年数・全間取り・全立地の借家平均）

両者を直接比較しない。SUUMO がある駅は SUUMO 優先、無い駅は政府 baseline を表示するハイブリッド方針で UI 整合性を取る。

#### 次回（5 年後 = 2028 年）の更新手順

1. e-Stat で次回調査（令和10年 = 2028 年）データ公開を確認
2. `scripts/build_government_rent_data.py` の `surveyYears` パラメータを `'2028'` に変更
3. 両スクリプトを再実行
4. JSON 上書き、UI 自動反映

## 過去に検討して却下した案

| 案 | 却下理由 |
|---|---|
| SUUMO 自動 scraping（クローラ） | TOS 違反、IP ban リスク、月次保守地獄。アフィリエイト提携してても改善せず |
| 公式 API（リクルート住まいカンパニー） | 公開 API 無、過去あった「Suumo Web Services」は 2017 年廃止 |
| A8 商品データフィード | SUUMO の case は deep link URL のみ、家賃データなし |
| 直接 B2B 契約 | 個人開発者には実質開かれていない（大手メディアと数百万円契約） |
| LLM batch のみ（政府データを使わない） | 学習データが古く市況反映に難。fallback としてのみ採用 |
